1. DISEÑO Y ARQUITECTURA
--------------------------------------------------------------------------------
* Cambios respecto al diseño inicial:
  - Implementación de arquitectura cross-platform real (Web y Móvil).
  - Migración a Google Identity Services (GIS) para soporte de login en navegadores.
  - Adaptación del sistema de archivos para usar XFile en lugar de dart:io (Web compliance).
  - Inclusión de controladores de inicialización asíncrona para evitar errores de carga.

* Modelo de datos definitivo:
  Tabla: incidencias
  - id (INT, Primary Key, Auto-increment)
  - uid_usuario (VARCHAR 255): ID único de Firebase (Google/Email).
  - categoria (VARCHAR 100): Alumbrado, Limpieza, Mobiliario, Vías.
  - descripcion (TEXT): Explicación del problema.
  - foto_url (VARCHAR 255): Enlace público a la imagen almacenada.
  - fecha (DATETIME): Fecha y hora del reporte.

* Explicación de la arquitectura implementada:
  - Frontend: Flutter (Dart) con enfoque en componentes desacoplados.
  - Autenticación: Firebase Auth (Email/Password y Google Sign-In).
  - Backend: Servidor PHP para procesamiento de archivos y lógica de negocio.
  - Base de Datos: MySQL (Relacional) para almacenamiento persistente.
  - Comunicación: Protocolo HTTP (MultipartRequest) con soporte CORS.

* API Definida:
  - POST /save_incidencia.php: Endpoint principal para recibir reportes con imagen.
    Campos: uid_usuario, categoria, descripcion, foto (file).
